# UNDRR syndication search widget

The **UNDRR syndication search widget** lets users search and filter content across UNDRR domains — undrr.org, preventionweb.net, and related sites. It provides keyword search, faceted filtering, and multiple display modes (list, card, or book card) in a single embeddable component.

> **Platform dependency:** This component requires access to the UNDRR search backend via `/search-endpoint`. It includes domain configurations, content type mappings, and taxonomy endpoints specific to the UNDRR platform. It is **not** a general-purpose search component.

> **Future:** A standalone, configurable search component without platform dependencies may be developed separately for broader use cases.

## Features

- Keyword search with responsive input — results update as you type
- Non-blocking result updates keep the page interactive while results load
- Removable filter chips for active filters
- Full ARIA support, keyboard navigation, screen reader announcements
- Styled with Mangrove design tokens
- Left-slide filter drawer on mobile, inline sidebar on desktop
- Custom select dropdowns (searchable, multi-select) with no external dependencies like Chosen.js

> **Implementation note (for contributors):** The non-blocking search experience is powered by React 19's concurrent rendering features: `useDeferredValue` keeps the search input responsive while results load in the background; `useTransition` marks filter changes as non-urgent to prevent UI blocking.

## Custom filters

`customFilters` are raw Elasticsearch query_string filters applied invisibly to all searches.
Users cannot see or remove these filters - they constrain the search scope.

**This example uses:**
```js
customFilters: [
  'type:(news OR publication)',      // Only news and publications
  'published_at:[2023-01-01 TO *]',  // Content from 2023 onwards
]
```

<Canvas of={FiltersStories.CustomFilters} meta={FiltersStories} />

**Common filter patterns:**
- `type:news` - Restrict to content type
- `type:(news OR publication)` - Multiple types (OR)
- `published_at:[2023-01-01 TO *]` - Date range
- `field_domain_access:www_preventionweb_net` - Domain filter
- `field_themes:123` - Taxonomy filter

## Custom facets

`customFacets` create user-selectable dropdown filters in the sidebar.
Each facet maps labels to Elasticsearch queries.

**This example uses:**
```js
customFacets: [
  {
    id: 'region',
    title: 'Region',
    multiSelect: false,
    options: [
      { label: 'Africa', query: 'field_regions:1' },
      { label: 'Americas', query: 'field_regions:2' },
      // ...
    ],
  },
  {
    id: 'hazard_type',
    title: 'Hazard Type',
    multiSelect: true,  // Multiple selections allowed
    options: [
      { label: 'Earthquake', query: 'field_hazards:101' },
      { label: 'Flood', query: 'field_hazards:102' },
      // ...
    ],
  },
]
```

<Canvas of={FiltersStories.CustomFacets} meta={FiltersStories} />

**Structure:**
```json
{
  "id": "unique_id",
  "title": "Display Title",
  "weight": 10,
  "multiSelect": false,
  "options": [
    { "label": "Option Label", "query": "field_name:value" }
  ]
}
```

| Property | Type | Description |
|----------|------|-------------|
| `id` | string | Unique identifier for the facet |
| `title` | string | Label shown in the sidebar |
| `weight` | number | Sort order (lower = higher) |
| `multiSelect` | boolean | Allow multiple selections |
| `options` | array | Array of `{label, query}` pairs |

## Query syntax reference

The search input uses Elasticsearch `query_string` syntax, forwarded directly to the `/search-endpoint` proxy. Drop any of the queries below into the search box on the [Default story](?path=/story/components-syndicated-search--default) to verify behaviour, or use them as a reference when writing `customFilters`.

### Basic queries

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `climate change` | Simple multi-word | Matches docs with both words (AND), stop words removed, fuzziness added |
| `disaster risk reduction` | Common phrase | Stop words "the", "of" etc. removed for consistent matching |
| `The Draft Articles on the Protection` | With stop words | Same results as "Draft Articles Protection" |

### Exact phrase matching

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `"climate change"` | Exact phrase | Matches exact phrase, no fuzziness, no stop word removal |
| `"Draft Articles"` | Exact phrase | Finds documents with "Draft Articles" as consecutive words |
| `"disaster risk reduction"` | Exact phrase | Stricter matching than without quotes |

### Wildcards and fuzzy

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `ris?` | Single char wildcard | Matches "risk", "rise", etc. |
| `red*` | Multi-char wildcard | Matches "reduction", "reduce", "red", etc. |
| `evnt~1` | Fuzzy (edit distance 1) | Matches "event" (typo tolerance) |
| `earthquke~2` | Fuzzy (edit distance 2) | Matches "earthquake" |

### Boolean operators

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `flood AND drought` | AND operator | Both terms required |
| `flood OR drought` | OR operator | Either term matches |
| `flood NOT tsunami` | NOT operator | Excludes tsunami results |
| `-"Red Cross"` | Exclude phrase | Excludes exact phrase |
| `(UNDRR AND "New York")` | Grouped | Parentheses for grouping |

### Field-specific searches

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `title:sendai` | Search title only | Matches "sendai" in title field |
| `_language:en` | Language filter | English content only |
| `type:landing` | Content type | Landing pages only |
| `type:(news OR publication)` | Multiple types | News or publications |

### Date ranges

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `published_at:[2023-01-01 TO 2023-12-31]` | Date range | Content from 2023 |
| `published_at:[2020-01-01 TO *]` | Open range | Content from 2020 onwards |
| `published_at:[* TO 2022-12-31]` | Open range | Content before 2023 |

### Boosting

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `history^1.1 review` | Term boosting | "history" weighted 1.1x higher |
| `title:sendai^2 body:sendai` | Field boosting | Title matches weighted 2x |

### Edge cases verified by the query builder

| Query | Description | Expected behaviour |
|-------|-------------|--------------------|
| `a` | Too short | Should not trigger search (min 3 chars) |
| `   spaces   ` | Extra whitespace | Trimmed and searched as `spaces` |
| `:` | Special char only | Handled gracefully |
| `""` | Empty quotes | Handled gracefully |

### Query sanitisation (auto-fixed)

These malformed queries are automatically sanitised to prevent Elasticsearch errors:

| Query | Problem | Auto-corrected to |
|-------|---------|-------------------|
| `"local` | Unclosed quote | `local` |
| `local OR ` | Trailing operator | `local` |
| `AND climate` | Leading operator | `climate` |
| `flood OR` | Trailing OR | `flood` |
| `AND` | Standalone operator | (empty — no search) |
| `climate-` | Trailing special char | `climate` |

## Server-rendered / CDN usage

SyndicationSearchWidget supports [layered hydration](https://github.com/unisdr/undrr-mangrove/blob/main/docs/HYDRATION.md). Use data attributes on a container element to configure the widget from server-rendered HTML — all attributes are optional:

```html
<div data-mg-search-widget
  data-search-endpoint="/search-endpoint"
  data-results-per-page="50"
  data-default-query=""
  data-default-sort="relevance"
  data-display-mode="list"
  data-show-facets="true"
  data-show-search-box="true"
  data-show-results-count="true"
  data-show-active-filters="true"
  data-show-pager="true"
  data-enable-hash-sync="true"
  data-default-filters='[{"key":"_language","value":"en"}]'>
</div>
```

| Attribute | Default | Description |
|-----------|---------|-------------|
| `data-search-endpoint` | — | Elasticsearch proxy URL |
| `data-results-per-page` | — | Number of results per page |
| `data-debounce-delay` | — | Milliseconds before executing search |
| `data-min-search-length` | — | Minimum characters to trigger search |
| `data-default-query` | — | Initial search query |
| `data-default-sort` | — | Initial sort order |
| `data-display-mode` | — | `'list'`, `'card'`, or `'card-book'` |
| `data-grid-columns` | — | Grid columns for card modes (2-6) |
| `data-query-append` | — | Query string to append to all searches |
| `data-show-search-box` | — | Show search input |
| `data-show-results-count` | — | Show result count |
| `data-show-facets` | — | Show facets sidebar |
| `data-show-active-filters` | — | Show active filter chips |
| `data-show-pager` | — | Show pagination |
| `data-show-search-metrics` | — | Show debug metrics |
| `data-show-search-timer` | — | Show search execution time |
| `data-enable-hash-sync` | — | Sync search state to URL hash |
| `data-default-filters` | — | JSON array of `{key, value}` filter objects |
| `data-allowed-types` | — | JSON object restricting content type options |
| `data-require-image` | — | Only return results with images (`true`/`false`) |
| `data-interestingness-tiers` | — | JSON array of tier names (e.g., `'["promoted","announced"]'`) |
| `data-longevity-tiers` | — | JSON array of tier names (e.g., `'["year","longtime","always"]'`) |
| `data-custom-filters` | — | JSON array of custom filter strings (e.g., `'["type:news"]'`) |
| `data-custom-facets` | — | JSON array of custom facet objects |
| `data-visible-teaser-fields` | — | JSON array of teaser field names to show |

All attributes are optional. Only attributes present on the container are included in the config; the component uses its own defaults for anything missing.

```js
import createHydrator from "https://assets.undrr.org/static/mangrove/1.7.0/components/hydrate.js";
import SyndicationSearchWidget, { fromElement } from "https://assets.undrr.org/static/mangrove/1.7.0/components/SyndicationSearchWidget.js";
createHydrator({ selector: "[data-mg-search-widget]", component: SyndicationSearchWidget, fromElement });
```

See the [hydration guide](https://github.com/unisdr/undrr-mangrove/blob/main/docs/HYDRATION.md) for the full API and integration details.

> **Integration note:** This is a complex-tier component. `fromElement` extracts only the attributes present on the container — Drupal modules and other consumer wrappers can merge additional config (`customFacets`, `customFilters`, etc.) that is too complex to express as data attributes.

## Architecture

> **Implementation note (for contributors):** Uses React 19 concurrent features: `useDeferredValue` keeps input responsive while results load; `useTransition` marks filter changes as non-urgent; `Suspense` provides loading skeletons; Context + `useReducer` manages search state without external dependencies.

### Component structure

```
SyndicationSearchWidget
├── SearchProvider (context)
│   ├── SearchForm (input + button)
│   ├── ActiveFilters (filter chips)
│   ├── SearchResults (with Suspense)
│   │   └── ResultItem (×n)
│   ├── FacetsSidebar (desktop)
│   │   ├── SortOptions (relevance/newest/oldest)
│   │   ├── FacetSelect (×n, from Elasticsearch aggregations)
│   │   │   └── SelectDropdown (custom Chosen-like component)
│   │   └── CustomFacetSelect (×n, editor-defined)
│   └── MobileFilterDrawer (mobile, <800px)
│       └── FacetsSidebar (reused)
```

### Custom select dropdown

The facet dropdowns use a custom `SelectDropdown` component instead of Chosen.js, with the same core functionality and no external dependencies:

| Feature | Description |
|---------|-------------|
| **Searchable** | Filter input shown when options > 8 |
| **Multi-select** | Checkbox indicators, multiple selections |
| **Single-select** | Radio indicators, one selection |
| **Keyboard navigation** | Arrow keys, Enter, Escape |
| **Touch-friendly** | 44px trigger height, 40px option height |
| **Accessible** | ARIA listbox pattern, screen reader support |

This means the component works in any environment (Drupal, Storybook, standalone) without requiring Chosen.js to be loaded.

### Mobile filter drawer

On screens ≤800px, the sidebar is replaced with a left-slide drawer:

- Amazon-style slide-in from the left edge
- Click backdrop to close (right side)
- Sticky header and footer with "View results" button
- Focus trapping for accessibility

UX patterns informed by [Pencil & Paper's mobile filter analysis](https://www.pencilandpaper.io/articles/ux-pattern-analysis-mobile-filters).

## Changelog

- **1.3.1**, 2026-05-23 ([mangrove#1031](https://github.com/unisdr/undrr-mangrove/issues/1031)): Fixed programmatic default filters being dropped on facet-less widgets. `buildQuery` skipped `post_filter` entirely when the facet UI was hidden, so any `defaultFilters` (or filters injected via the `configModifiers` hook) never reached Elasticsearch and the widget returned an unfiltered search. `post_filter` now applies whenever there are active facet or custom-facet values, even with facets hidden; aggregations stay skipped, so the 1.2.1 payload optimization is unchanged. Regression introduced in 1.2.1.
- **1.3.0** — 2026-05-07 ([mangrove#981](https://github.com/unisdr/undrr-mangrove/issues/981)):
  - New `facets` config prop (`false | 'sidebar' | 'horizontal'`, default `'sidebar'`) replaces the boolean `showFacets` for layout control; `'horizontal'` renders the facets as a strip above the results region instead of a sidebar. Legacy `showFacets: false` continues to work as a backwards-compatible alias for `facets: false`. Drupal sites can pass the layout via `data-facets="horizontal"`
  - New `facetsTarget` config prop (CSS selector) renders the facets UI into an external DOM region via React portal — letting consuming pages place facets in a left rail or other region outside the widget's own container while results render where the widget is mounted. Search context spans the portal so facets and results stay in sync without extra wiring. Drupal sites pass the selector via `data-facets-target`. If the selector doesn't match an element at mount time, the widget falls back to the in-widget `facets` layout and logs a console warning
  - New `searchTarget` config prop (CSS selector) renders the search input into an external DOM region via React portal — mirrors `facetsTarget`. Used together they let consumers freely arrange search input / facets / results across page regions while keeping a single widget instance and shared `SearchContext`. Drupal sites pass the selector via `data-search-target`. Falls back to in-widget rendering (with a console warning) if the target element is not present at mount time. For **cross-page** hero patterns the recommended approach is a plain HTML `<form action="/search">` submitting to the results page with a `#query=` fragment
  - Storybook stories regrouped under `Components/Syndicated search/{Widget,Bar}` and split into category files (`Layouts`, `Display modes`, `Filters`, `Integrations`, `Toggles`) so the sidebar reflects the family rather than a flat list of 26 entries. Pure reorganisation — no story content removed, just moved. The `QuerySyntax` and `StaticMockup` reference stories are now reference sections in this MDX page
  - `SearchForm` submit and clear buttons now use the standard `mg-button mg-button-primary` and `mg-icon-search` / `mg-icon-close` primitives instead of a bespoke variant with an inline Lucide SVG; `.mg-search__input` aligns to the same padding, border-width and border-radius tokens as `mg-button` so the input and submit button share the same natural height and corner radius; clear button is a 20×20 hit target with darker default color and brand-color hover
  - Removed the in-input loading spinner (`.mg-search__loading`) and the submit-button spinner (`.mg-search__submit-spinner` / `.mg-search__submit--loading`) so loading state is conveyed by a single indicator — the widget-level progress strip plus `aria-busy` on the results region; the submit button now keeps a stable "Search" label and search icon while a search runs
  - Fixed URL hash query being stomped on initial mount — the `deferredQuery` effect was unconditionally dispatching `setQuery('')` on first render, racing `useHashSync`'s hash-loaded query before INITIALIZE could preserve it; the effect now skips its first render
