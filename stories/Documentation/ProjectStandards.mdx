# About Mangrove

Mangrove defines shared content structures for UNDRR web properties. A card has an image, a title, a link, and summary text. A hero has a heading, a subline, and an optional call to action. A footer pulls in shared links. These content patterns are what Mangrove provides.

Everything else stays yours: brand colors, component treatments, UI framework, application-specific interfaces like forms, data workflows, and dashboards. Properties share a visual baseline (typography, interaction patterns, element styling) but control their own color identity and surface treatments.

If your project uses those same content structures, everything that depends on them works: syndicated content renders correctly, accessibility fixes propagate, and content updates don't break across sites. Your project won't look identical to undrr.org -- the result is shared behavior and structure, not a uniform straitjacket.

## Content structure vs. visual style

Think about a card component. The content structure is: image, title, link, summary text, maybe a date and a content type tag. That structure is what Mangrove defines. The visual treatment -- rounded corners or square, drop shadow or flat, serif or sans-serif -- is up to your theme.

You can redesign how a card looks without touching Mangrove. You cannot remove the link from a card without breaking every content feed that expects cards to have links.

Content structure is the hard part. Changing CSS is a few variables. Changing what a component carries is a breaking change that affects every site using it.

The same principle applies to the mega menu (sections, groups, links are shared; visual treatment is per-theme), the footer (content slots are shared; layout can vary), and every other shared component. If you need to rethink how something looks, that's a conversation with your designer. If you need to rethink what content a component carries, that's a conversation with the Mangrove team, because it affects everyone.

## What stays the same, what doesn't

Across all themes:

- Content structure of shared components (what a card, hero, or footer contains)
- Navigation patterns (mega menu sections, footer link groups, logo placement)
- Interaction behavior (hover states, focus rings, transitions)

What each property controls:

- Brand colors and interactive color treatments
- Component border radii
- Button styles, tag colors, tab appearance
- Hero overlays and accent treatments
- Choice of application-specific UI library (data tables, forms, dashboards)

## What Mangrove is not

**Mangrove is not a visual brand enforcer.** It doesn't dictate colors, button styles, or how your hero looks. Those are design tokens (named values for colors, spacing, and component treatments) that each property controls. You can make the title bigger, the subline smaller, the accent color green. That's all CSS. Mangrove only cares that the hero *has* a title, a subline, and a link, because if a hero stops carrying links, every content pattern that assumed it would breaks.

**Mangrove is not a full application framework.** It doesn't handle data tables, form validation, dashboards, workflow state, or user management. If you need PrimeReact or Material UI for that, use them. Mangrove sits alongside those libraries.

**Mangrove is not heavy custom code.** It's a standard React codebase (React, Sass, Storybook). Someone who knows React can read, modify, and extend any component. There's no proprietary framework, no code generation, no specialized toolchain. The only custom part is the content patterns, which are specific to UNDRR's content model. The library is open source, published on npm, and documented here. If the team changes, the next people inherit familiar technology, not a black box.

## Why this approach

UNDRR is a small organization with a smaller web team supporting many properties. Before Mangrove, PreventionWeb, MCR, IRP, and others each maintained their own UI. The same bug got fixed three times on three platforms. That alone would justify a shared library.

But the stronger reason is content. UNDRR publishes news, publications, and partner listings in Drupal. That content gets syndicated across sites as cards, search results, and content feeds, all with `mg-` CSS class names baked into the markup. If your property doesn't use the same content structure, that syndicated content renders unstyled or not at all.

And when UNDRR rebrands -- which will happen eventually -- properties using Mangrove update a CSS import. Properties with their own content components do that work alone. Work that any property puts into Mangrove benefits every other property that uses it. (The [shared infrastructure](#shared-infrastructure) section below describes the specific services that depend on this alignment.)

## Mangrove alongside application frameworks

If your team has already invested in PrimeReact or a similar library, that investment isn't wasted. Mangrove sits alongside application-level libraries. It doesn't replace them.

| Layer | Owned by | Examples |
|-------|----------|----------|
| **Brand and content** | Mangrove | Header, footer, mega menu, content cards, typography, color tokens, syndicated content blocks |
| **Application UI** | Your framework | Data tables, form controls, dashboards, filters, tree views, date pickers |

A page might look like this (this example assumes a React app importing Mangrove from npm; for Drupal integration, see the <LinkTo kind="getting-started-a-getting-started-guide" story="docs">getting started guide</LinkTo>):

```jsx
// Mangrove handles the UNDRR content shell
import { MegaMenu } from '@undrr/undrr-mangrove';
// Your app library handles data-heavy interfaces
import { DataTable } from 'primereact/datatable';
import { Column } from 'primereact/column';

function CountryDashboard({ navSections, riskData }) {
  return (
    <>
      <MegaMenu sections={navSections} />
      <main>
        <DataTable value={riskData} paginator rows={10}>
          <Column field="country" header="Country" />
          <Column field="riskIndex" header="Risk index" />
          <Column field="lastUpdated" header="Last updated" />
        </DataTable>
      </main>
    </>
  );
}
```

The two don't conflict. Mangrove scopes all styles under the `mg-` BEM prefix, and its React components externalize `react` and `react-dom` so there's no runtime duplication. PrimeReact, Material UI, Bootstrap, Tailwind -- these libraries handle generic UI patterns well, but they don't carry the content patterns UNDRR uses (how cards are structured, how the mega menu behaves, how syndicated content renders across sites). Mangrove adds that content layer. The two solve different problems at different layers, so keeping both is the intended setup.

## Why semantic CSS

Mangrove uses BEM-namespaced CSS (Block-Element-Modifier naming like `mg-card`, `mg-hero`, `mg-footer__links`) rather than utility classes. The reason comes down to content portability.

UNDRR syndicates content as rendered HTML with `mg-` class names baked into the markup. That content gets displayed across PreventionWeb, MCR2030, IRP, and other properties, each with its own visual identity. With semantic class names, the HTML says *what something is* (`mg-card`), and the CSS says *how it looks*, per theme. Swapping visual identity is a CSS import, not a template audit.

Component libraries like PrimeReact also use semantic naming (`p-card`, `p-card-header`) and have their own theming systems. But no component library covers everything -- there's no PrimeReact hero, no off-the-shelf mega menu that matches your site's navigation model. Developers fill those gaps, and when they reach for utility classes to do it, the visual decisions get baked into the markup: `bg-stone-50 rounded-xl p-8 flex flex-col gap-4`. Which of those classes are structural? Which are decorative? When the next property needs a different look, there's no reliable way to separate them.

This is a [well-documented tradeoff](https://bradfrost.com/blog/post/the-many-faces-of-themeable-design-systems/). Utility-first CSS is faster for building single-brand sites. But when the same content needs to render across multiple visual contexts, organizations consistently reach for semantic naming.

Mangrove is not an outlier here. Within the UN system alone, OCHA built [Common Design](https://github.com/UN-OCHA/common_design) with `cd-` prefixed BEM classes to share components across ReliefWeb, GHO, and other humanitarian sites. UNDP, UNICEF, WFP, FAO, and ILO have all built their own design systems to solve the same multi-site consistency problem. Outside the UN, the European Commission's [Europa Component Library](https://ec.europa.eu/component-library/) uses `ecl-` prefixed BEM to serve both EC and EU websites from the same HTML with different CSS distributions -- the same architecture Mangrove uses with its four theme files. Shopify went even further: in 2025 they [deprecated Polaris React in favor of web components](https://www.shopify.com/partners/blog/polaris-unified-and-for-the-web) to make their admin components framework-agnostic, while keeping their semantic token architecture (`--p-color-*`, `--p-space-*`) intact. They still recommend Tailwind for single-brand storefronts where portability isn't a concern.

For further reading, see [Design token-based UI architecture](https://martinfowler.com/articles/design-token-based-ui-architecture.html) on martinfowler.com.

## Adopting Mangrove

Not every component in every system needs to come from Mangrove. Here's how to identify what does.

**Displaying UNDRR content** (publications, news, search results, partner listings): you need Mangrove CSS and the relevant content components. Without them, syndicated content renders as unstyled HTML.

**Standalone tool with no shared content**: Mangrove design tokens (SCSS variables for colors, spacing, typography) give you brand consistency with zero component lock-in. Or skip Mangrove entirely and use your own styling. The content structure matters more than the CSS.

**Mixed: domain-specific UI plus shared content** (custom data tables alongside syndicated news feeds): use whatever framework handles your application patterns best. Use Mangrove for the shared layer -- navigation, cards, content blocks. Both coexist by design.

**Open source tool deployed by organizations outside UNDRR**: the architecture supports external deployments without UNDRR branding. Your UNDRR-hosted instance can add the Mangrove header, footer, and content feeds as a separate layer, while other deployments use whatever styling they choose. Whether your specific tool should ship with or without UNDRR branding is a decision to make with the communications team. The technical point is that Mangrove's content structures are independent of the theme, so you can maintain one codebase that works with or without the UNDRR layer.

### Incremental adoption

You don't have to integrate everything at once:

1. Add the Mangrove CSS (via CDN or npm)
2. Swap in the header and footer
3. Use Mangrove cards where you display syndicated content
4. Adopt design tokens in your own SCSS

The rest of your application stays as-is. There's no expectation that you will eventually adopt all of Mangrove. Step 1 or 2 may be your permanent integration level, and that's fine.

### Creating a theme

A Mangrove theme can be as simple as a single SCSS variable override file. The existing theme files are small:

- PreventionWeb: 48 lines of variable overrides
- IRP: 50 lines
- MCR2030: 65 lines

The overridable token set currently covers interactive colors, button and tag treatments, hero styling, form controls, and tab appearance. Foundational tokens like typography and spacing are shared across all themes. If you need to customize something outside that set, talk to the Mangrove team -- the overridable set has grown over time as properties have needed more flexibility. Start with the <LinkTo kind="getting-started-theming-guide" story="docs">theming guide</LinkTo>, which walks through creating a new theme file from scratch.

### Bundle sizes

The full CSS theme file is ~26 KB gzipped. If your application already uses React, each component adds roughly 1-15 KB gzipped depending on complexity (React and ReactDOM are externalized, so there's no runtime duplication). Mangrove also works without React -- the CSS and HTML patterns can be used in any server-rendered template (PHP, Twig, etc.) with no JavaScript dependency.

For integration details, see the <LinkTo kind="getting-started-a-getting-started-guide" story="docs">getting started guide</LinkTo>, <LinkTo kind="getting-started-sass-integration" story="docs">Sass integration guide</LinkTo>, or <LinkTo kind="getting-started-react-integration" story="docs">React integration guide</LinkTo>.

### When something doesn't fit

Some real constraints may require working outside Mangrove:

- Legacy systems that can't load Mangrove CSS
- Proven performance issues with specific components
- Conflicting dependencies
- Specialized interfaces or domain-specific workflows
- Critical launch deadlines (with a plan to add Mangrove support later)

If you hit one of these, talk to the Mangrove team. Often there's a lighter integration path. If there isn't, document the constraint and plan to revisit it.

## Shared infrastructure

Content syndication is where alignment with Mangrove is most visible, but it's one of several shared services. These are maintained once for every connected property:

**Syndicated search.** The SyndicationSearchWidget queries UNDRR's Elasticsearch index and can render results as cards, lists, or book covers. On a landing page, you can strip out the search box, facets, and pager entirely and just show a row of content cards -- a "latest publications" rail or a topic-specific news feed:

```jsx
<SyndicationSearchWidget config={{
  searchEndpoint: 'https://www.undrr.org/search-endpoint',
  displayMode: 'card',
  resultsPerPage: 4,
  showSearchBox: false,
  showFacets: false,
  showPager: false,
  defaultSort: 'newest',
  customFilters: ['type:publication'],
}} />
```

The cards use `mg-` classes. Without Mangrove CSS, they're unstyled divs. See the <LinkTo kind="components-syndicationsearchwidget" story="syndicated-cards">syndicated cards example</LinkTo> for a live demo.

**Footer syndication.** The Footer component pulls UNDRR's global footer content dynamically from PreventionWeb's syndication widget. Footer links, legal notices, and social media icons stay consistent across properties without each site maintaining its own copy.

**Cookie consent.** The CookieConsentBanner loads UNDRR's cookie configuration from a shared CDN (`assets.undrr.org/static/cookie-banner/v1/`). The configuration is maintained once and every property picks it up.

**Critical messaging.** When UNDRR needs to broadcast an urgent message across all properties -- service outages, emergency announcements -- the critical messaging system injects a message bar via a single script tag. It's hosted on GitHub Pages (independent of Drupal) and supports location targeting and severity levels. Properties outside the shared infrastructure miss these broadcasts.

**Analytics.** A shared GA4 enhancement script auto-detects the current domain and configures the correct property ID and cross-domain measurement. No per-site configuration needed.

## Accessibility

Accessibility is mandatory for every UNDRR digital product, whether it's a content site or an application. Mangrove targets WCAG 2.1 AA. Components with test suites include jest-axe automated checks, and the team manually verifies screen reader support, keyboard navigation, and color contrast. Coverage is expanding; see the <LinkTo kind="getting-started-accessibility" story="docs">accessibility guide</LinkTo> for the testing methodology and the <LinkTo kind="getting-started-component-contribution-guide" story="docs">contribution guide</LinkTo> for testing requirements applied to new components.

If you use Mangrove for the content layer, you inherit that testing for every component you adopt. If you build your own content components, you own compliance for each one separately.

## Contributing and governance

The UNDRR web team maintains Mangrove. Breaking changes follow semantic versioning, and the team documents them in release notes. Properties can pin to a specific version via npm or CDN to control when they adopt updates.

If Mangrove doesn't have what you need, extend it. File a request as a [GitHub issue](https://github.com/unisdr/undrr-mangrove/issues) describing the use case, or build it yourself and contribute it back. When your team builds something reusable -- a country selector, a progress tracker -- contributing it upstream means it gets maintained alongside the rest of the system instead of staying in one project. See the <LinkTo kind="getting-started-component-contribution-guide" story="docs">component contribution guide</LinkTo>.

For architectural questions or time-sensitive needs, contact the UNDRR web team directly.
