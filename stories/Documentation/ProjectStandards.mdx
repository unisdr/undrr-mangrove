# Mangrove: the UNDRR standard

Mangrove is UNDRR's shared design system -- a common component structure for all web properties, where each property keeps its own visual identity.

If you're building a UNDRR property and wondering whether you need another component library on top of your existing stack, this page explains what Mangrove handles, what it doesn't, and why it matters even if your project is an application rather than a content website. If you're new to Mangrove and just want to get started, head to the <LinkTo kind="getting-started-a-getting-started-guide" story="docs">getting started guide</LinkTo>.

## What Mangrove covers (and what it does not)

Mangrove provides shared components -- navigation, headers, footers, content cards, syndicated content blocks. Each property supplies its own design tokens for colors, spacing, typography, and border radii. Your application doesn't have to look like undrr.org -- it has to use the same component markup so that shared content, accessibility patterns, and updates work everywhere.

In practice, the split looks like this. What stays the same across all themes:

- Architecture of shared component patterns (cards, heroes)
- Navigation patterns (mega menu layout, footer structure, logo placement)
- Interaction patterns (hover states, transitions, focus rings)

What each property controls:

- Color palette and brand colors
- Spacing scale and border radii
- Button styles, tag colors, tab appearance
- Hero overlays and accent treatments
- Choice of application-specific UI library (data tables, forms, dashboards, etc.)

Mangrove doesn't try to be a general-purpose application framework. Data tables, form validation, complex dashboards, workflow state, user management -- those are outside its scope.

## Why it matters for your project

UNDRR is a small organization with a smaller web team supporting many properties. We can't afford for each website to operate as its own vertical with its own component stack. Before Mangrove, that's what we had -- PreventionWeb, MCR, IRP, and others each maintaining their own UI. Every accessibility fix, rebrand, or onboarding workflow had to happen multiple times. The same bug got fixed three times on three platforms.

That's the organizational background. Here's what it means for your project.

UNDRR will rebrand again. When that happens, Mangrove-based properties update a single CSS import. A property with its own branding components does that work alone. Same story when a hero layout changes -- Mangrove components get updated and tested once, but if you built your own navigation or card components, you audit each of those separately.

There's also the content problem. UNDRR publishes news, publications, and partner listings in Drupal with `mg-` CSS class names baked in. If your property doesn't load Mangrove's CSS, that syndicated content renders as unstyled fragments. And shared services like emergency broadcasts, cookie consent, and analytics configuration only reach properties that are connected to the shared infrastructure.

The work from your project to support Mangrove supports every other property, and the work they do supports yours.

## Mangrove alongside application frameworks

If your team has already invested in PrimeReact or a similar library, that investment is not wasted. Mangrove is meant to sit alongside application-level libraries, not replace them.

| Layer | Owned by | Examples |
|-------|----------|----------|
| **Brand and content** | Mangrove | Header, footer, mega menu, content cards, typography, color tokens, syndicated content blocks |
| **Application UI** | Your framework | Data tables, form controls, dashboards, filters, tree views, date pickers |

A page might look like this:

```jsx
// Mangrove handles the UNDRR brand shell
import { MegaMenu } from '@undrr/undrr-mangrove';
// Your app library handles data-heavy interfaces
import { DataTable } from 'primereact/datatable';
import { Column } from 'primereact/column';

function CountryDashboard({ navSections, riskData }) {
  return (
    <>
      <MegaMenu sections={navSections} />
      <main>
        <DataTable value={riskData} paginator rows={10}>
          <Column field="country" header="Country" />
          <Column field="riskIndex" header="Risk index" />
          <Column field="lastUpdated" header="Last updated" />
        </DataTable>
      </main>
    </>
  );
}
```

The two don't conflict. Mangrove scopes all styles under the `mg-` BEM prefix, and its React components externalize `react` and `react-dom` so there's no runtime duplication. A common concern is that mixing libraries creates visual inconsistency, but the inconsistency Mangrove prevents is between your application and every other UNDRR property, not within it.

### "But my component library has a larger community"

True -- whether it's PrimeReact, Material UI, Bootstrap, or Tailwind, these libraries have broad community support for generic UI patterns. But they don't carry the content patterns UNDRR actually uses: how cards are structured, how the mega menu behaves, how syndicated content renders across sites, how the footer pulls in shared content from PreventionWeb. Those are UNDRR-specific problems, and Mangrove is where they get solved once instead of separately on every property.

## Adopting Mangrove

Not every component in every system needs to come from Mangrove. Here's how to identify what does.

**If your system displays any content from UNDRR Drupal sites** (publications, news, search results, partner listings): you need Mangrove CSS and the relevant brand components. Without them, syndicated content renders broken.

**If your system is a standalone application with no shared content**: Mangrove design tokens (SCSS variables for colors, spacing, typography) give you brand consistency with zero component lock-in.

**If your application has domain-specific UI needs** (custom data tables, specialized filters, workflow dashboards): use whatever framework handles those patterns best. Use Mangrove for the shared layer -- navigation, cards, accordions, content blocks. Both coexist by design.

### Incremental adoption

You don't have to integrate everything at once:

1. Add the Mangrove CSS (via CDN or npm)
2. Swap in the header and footer
3. Use Mangrove cards where you display syndicated content
4. Adopt design tokens in your own SCSS

The rest of your application stays as-is.

### Creating a theme

A Mangrove theme can be as simple as a single SCSS variable override file or as detailed as you need it to be. The existing theme files are small:

- PreventionWeb: 48 lines of variable overrides
- IRP: 50 lines
- MCR2030: 65 lines

A new theme for your application would be similar -- your color palette, spacing preferences, and any component-level tweaks. Your application keeps its own visual identity while UNDRR content renders correctly inside it. Start with the <LinkTo kind="getting-started-theming-guide" story="docs">theming guide</LinkTo>, which walks through creating a new theme file from scratch.

### Bundle sizes

The full CSS theme file is ~26 KB gzipped. React components are ES modules with React and ReactDOM externalized, so if your application already uses React, each component adds roughly 1-15 KB gzipped depending on complexity. Mangrove also works without React -- the CSS and HTML patterns can be used in any server-rendered template (PHP, Twig, etc.) with no JavaScript dependency.

For integration details, see the <LinkTo kind="getting-started-a-getting-started-guide" story="docs">getting started guide</LinkTo>, <LinkTo kind="getting-started-sass-integration" story="docs">Sass integration guide</LinkTo>, or <LinkTo kind="getting-started-react-integration" story="docs">React integration guide</LinkTo>.

### When something doesn't fit

Some real constraints may require working outside Mangrove:

- Legacy systems that can't load Mangrove CSS
- Proven performance issues with specific components
- Conflicting dependencies
- Specialized interfaces or domain-specific workflows
- Critical launch deadlines (with a plan to add Mangrove support later)

If you hit one of these, talk to the Mangrove team. Often there's a lighter integration path that works. If there genuinely isn't, document the constraint and plan to revisit it.

## Shared infrastructure

Content syndication is where alignment with Mangrove is most visible, but it's one of several shared services. These are maintained once for every connected property:

**Syndicated search.** The SyndicationSearchWidget queries UNDRR's Elasticsearch index and can render results as cards, lists, or book covers. On a landing page, you can strip out the search box, facets, and pager entirely and just show a row of content cards -- a "latest publications" rail or a topic-specific news feed:

```jsx
<SyndicationSearchWidget config={{
  searchEndpoint: 'https://www.undrr.org/search-endpoint',
  displayMode: 'card',
  resultsPerPage: 4,
  showSearchBox: false,
  showFacets: false,
  showPager: false,
  defaultSort: 'newest',
  customFilters: ['type:publication'],
}} />
```

The cards use `mg-` classes. Without Mangrove CSS, they're unstyled divs. See the <LinkTo kind="components-syndicationsearchwidget" story="syndicated-cards">syndicated cards example</LinkTo> for a live demo.

**Footer syndication.** The Footer component pulls UNDRR's global footer content dynamically from PreventionWeb's syndication widget. Footer links, legal notices, and social media icons stay consistent across properties without each site maintaining its own copy.

**Cookie consent.** The CookieConsentBanner loads UNDRR's cookie configuration from a shared CDN (`assets.undrr.org/static/cookie-banner/v1/`). The configuration is maintained once and every property picks it up.

**Critical messaging.** When UNDRR needs to broadcast an urgent message across all properties -- service outages, emergency announcements -- the critical messaging system injects a message bar via a single script tag. It's hosted on GitHub Pages (independent of Drupal) and supports location targeting and severity levels. Properties outside the shared infrastructure miss these broadcasts.

**Analytics.** A shared GA4 enhancement script auto-detects the current domain and configures the correct property ID and cross-domain measurement. No per-site configuration needed.

## Accessibility

Accessibility is mandatory for every UNDRR digital product, whether it's a content site or an application. Mangrove targets WCAG 2.1 AA. Components are tested with jest-axe for automated checks and manually verified for screen reader support, keyboard navigation, and color contrast. See the <LinkTo kind="getting-started-accessibility" story="docs">accessibility guide</LinkTo> for the testing methodology.

If you use Mangrove for the brand layer, you inherit that testing for every component you adopt. If you build your own branding components, you own compliance for each one separately.

## Contributing and governance

Mangrove is maintained by the UNDRR web team, who own component development, documentation, accessibility compliance, and breaking-change decisions. Breaking changes follow semantic versioning and are documented in release notes.

If Mangrove doesn't have what you need, extend it. File a request as a [GitHub issue](https://github.com/unisdr/undrr-mangrove/issues) describing the use case, or build it yourself and contribute it back. When your team builds something reusable -- a country selector, a progress tracker -- contributing it upstream means it gets maintained alongside the rest of the system instead of staying siloed in one project. See the <LinkTo kind="getting-started-component-contribution-guide" story="docs">component contribution guide</LinkTo> for the full process.

For architectural questions or time-sensitive needs, contact the UNDRR web team directly.
