# AI and MCP integration

Mangrove publishes machine-readable component metadata alongside its Storybook documentation, so AI coding agents can discover and use components accurately. This guide covers how agents find Mangrove, what data they get, and how you can help.

## How AI agents discover Mangrove

Mangrove provides a three-tier discovery system designed for how AI agents actually work. Agents that can fetch URLs will find structured data at well-known paths; agents working from the repository will find the same information in source.

### Tier 1: `llms.txt` (entry point)

The deployed Storybook site serves a [`llms.txt`](https://unisdr.github.io/undrr-mangrove/llms.txt) file at the site root. This follows the [llms.txt convention](https://llmstxt.org/) — a plain-text discovery file that tells AI agents what the project is, where to find documentation, and where to find the machine-readable component index.

An agent fetching `https://unisdr.github.io/undrr-mangrove/llms.txt` gets:

- Project description and key conventions (CSS prefix, BEM naming, themes)
- Links to documentation, repository, and npm
- A pointer to the component index at `ai-components/index.json`

### Tier 2: component index (`ai-components/index.json`)

The [component index](https://unisdr.github.io/undrr-mangrove/ai-components/index.json) is a lightweight JSON file (~29 KB) listing all components with their names, descriptions, import statements, and documentation URLs. Each entry includes a `detailsUrl` pointing to the per-component JSON file.

```json
{
  "id": "components-pager",
  "name": "Pager",
  "description": "Pager component.",
  "import": "import { Pager } from \"@undrr/undrr-mangrove\";",
  "docsUrl": "https://unisdr.github.io/undrr-mangrove/?path=/docs/components-pager--docs",
  "detailsUrl": "https://unisdr.github.io/undrr-mangrove/ai-components/components-pager.json"
}
```

This gives agents enough context to identify the right component without fetching everything.

### Tier 3: per-component details (`ai-components/{id}.json`)

Each component has its own JSON file with full prop definitions, types, defaults, descriptions, and code examples extracted from Storybook stories. These files are typically 1-5 KB each.

An agent working on a Pager implementation would fetch `ai-components/components-pager.json` to get:

- All props with types, required/optional status, and defaults
- Code examples from every Storybook story variant
- A link to the full Storybook documentation page

### How agents should use this

The intended workflow for an AI agent:

1. Fetch `llms.txt` to understand the project
2. Fetch `ai-components/index.json` to find relevant components
3. Fetch `ai-components/{id}.json` only for the specific components it needs

This keeps token usage low. An agent building a single component fetches ~30 KB of index data plus ~3 KB for the component details, rather than the full 700+ KB Storybook manifest.

## Directing your AI agent to Mangrove

### Point agents to the repository

For agents that work with source code (Claude Code, Cursor, Copilot), the most reliable approach is to reference the GitHub repository. Agents can read source files directly:

```
I'm working with the UNDRR Mangrove component library.
Repository: https://github.com/unisdr/undrr-mangrove

Help me create a page layout using the MegaMenu, Hero, and Footer components.
```

The repository contains `CLAUDE.md` (for Claude Code) and `.github/copilot-instructions.md` (for GitHub Copilot) with full project context.

### Point agents to `llms.txt`

For agents that can fetch URLs but don't have direct file access:

```
Fetch the component library metadata from:
https://unisdr.github.io/undrr-mangrove/llms.txt

Use these components to build the requested UI.
```

### Point agents to a specific component

For the most accurate results, reference a specific component's detail file:

```
Fetch the Pager component metadata:
https://unisdr.github.io/undrr-mangrove/ai-components/components-pager.json

Create a similar paginated list using this component.
```

## Configuring AI coding tools

### Claude Code

Add the repository reference to your conversation, or if working in the Mangrove repo, Claude Code automatically reads `CLAUDE.md` for project context.

For projects that consume Mangrove, add to your `CLAUDE.md`:

```markdown
## Component library

This project uses the UNDRR Mangrove component library.
- AI manifest: https://unisdr.github.io/undrr-mangrove/llms.txt
- Component index: https://unisdr.github.io/undrr-mangrove/ai-components/index.json
- CSS prefix: mg-
- Naming: BEM (e.g., mg-card__title, mg-button--primary)
```

### Cursor

Add to your `.cursor/rules` or project instructions:

```
Use the UNDRR Mangrove component library for all UI components.
AI manifest: https://unisdr.github.io/undrr-mangrove/llms.txt
Component index: https://unisdr.github.io/undrr-mangrove/ai-components/index.json
CSS prefix: mg-
```

### GitHub Copilot

The repository includes `.github/copilot-instructions.md` with project conventions. For consuming projects, include a comment at the top of your file:

```jsx
// Using UNDRR Mangrove components
// Component index: https://unisdr.github.io/undrr-mangrove/ai-components/index.json
// CSS prefix: mg-, BEM naming
```

## Best practices for AI-assisted development

### Be specific about Mangrove

When prompting AI tools, mention Mangrove explicitly and include key conventions:

```
# Good prompt
Using the Mangrove component library (mg- prefix classes),
create a hero section with a primary CTA button.

# Less effective prompt
Create a hero section with a button.
```

### Provide context about conventions

Include these key facts in your prompts:

- **CSS prefix**: All classes use the `mg-` prefix (e.g., `mg-button`, `mg-card`)
- **BEM naming**: Classes follow BEM conventions (e.g., `mg-card__title`, `mg-button--primary`)
- **Semantic HTML**: Components use proper HTML elements for accessibility
- **4 themes**: undrr (default), preventionweb, irp, mcr2030
- **RTL support**: Arabic and other RTL languages are supported

### Validate the output

AI-generated code should be validated against:

1. **Class naming**: All classes should use the `mg-` prefix
2. **HTML structure**: Match the documented component structure
3. **Accessibility**: Include proper ARIA attributes
4. **Responsiveness**: Follow Mangrove's responsive patterns

## For contributors: local MCP server

If you're developing Mangrove components locally, you can enable the full MCP (Model Context Protocol) experience. This provides a live connection between your AI agent and the running Storybook dev server.

> **Note**: This section is for contributors working on the Mangrove library itself, not for consumers using the components.

### What MCP provides

When running locally with the MCP addon, AI agents can:

- Query component information programmatically via API
- Get direct URLs to specific stories for visual verification
- Access real-time component documentation as you develop
- Validate generated components against interaction and accessibility tests

### Setting up MCP for local development

1. **Install the MCP addon**:
   ```bash
   yarn add -D @storybook/addon-mcp
   ```

2. **Add to your Storybook config** (`.storybook/main.js`):
   ```javascript
   addons: [
     // ... other addons
     '@storybook/addon-mcp',
   ],
   ```

3. **Start Storybook**:
   ```bash
   yarn dev
   ```

4. **Configure your AI tool** to connect to `http://localhost:6006/mcp`

   For Claude Code:
   ```bash
   claude mcp add storybook-mcp --transport http http://localhost:6006/mcp --scope project
   ```

See the [Storybook MCP documentation](https://storybook.js.org/blog/storybook-mcp-sneak-peek/) for detailed setup instructions for other AI tools.

## How the AI manifest is generated

The component metadata files are generated automatically during `yarn build`. The build script (`scripts/generate-ai-manifest.js`) transforms Storybook's internal components manifest (produced by the `experimentalComponentsManifest` feature) into the tiered AI-friendly format described above.

The generated files are deployed alongside the static Storybook site to GitHub Pages, so they stay in sync with the documentation and are always current with the latest published version.

To regenerate manually:

```bash
node scripts/generate-ai-manifest.js
```
