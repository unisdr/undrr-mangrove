## Component development workflow

### 1. Planning phase

Before starting development:

- Check if a similar component already exists to avoid duplication
- Define the component's purpose, functionality, and variants
- Identify the props/parameters the component will accept
- Consider accessibility requirements from the beginning
- Plan responsive behaviors for different screen sizes
- Ensure your component follows the shared content patterns described in **<LinkTo kind="getting-started-about-mangrove" story="docs">About Mangrove</LinkTo>**

### 2. Component structure

All components in Mangrove follow a consistent file structure:

```
stories/Components/[ComponentCategory]/[ComponentName]/
├── ComponentName.jsx              # React component implementation
├── ComponentName.mdx              # Documentation
├── ComponentName.stories.jsx      # Storybook stories
├── component-name.scss            # Component-specific styles (optional)
├── ComponentName.fromElement.js   # Hydration prop extraction (optional)
├── ComponentName.hydrate.js       # Barrel re-exporting component + fromElement (optional)
└── __tests__/
    ├── ComponentName.test.jsx
    └── ComponentName.fromElement.test.js  # Tests for prop extraction (optional)
```

The `fromElement` and `hydrate` files are needed when a component is used in server-rendered contexts (Drupal, Astro). See the [Hydration guide](https://github.com/unisdr/undrr-mangrove/blob/main/docs/HYDRATION.md) for when and how to add these files.

#### Component categories

Place your component in the appropriate category folder:

- `Buttons` - Interactive elements that trigger actions
- `Cards` - Container components that group related content
- `Forms` - Input components and form-related elements
- `Hero` - Hero banner components
- `Map` - Map-related components
- `MegaMenu` - Navigation menu components
- `Tab` - Tab components for organizing content
- `Charts` - Data visualization components

If none of the existing categories fit, discuss with the team before creating a new category.

## Code standards and conventions

### React component guidelines

1. **Use functional components with hooks**

   ```jsx
   import React from 'react';

   export function MyComponent({ prop1, prop2, ...props }) {
     // Implementation
     return (
       <div className="mg-my-component" {...props}>
         {/* Component content */}
       </div>
     );
   }
   ```

2. **Props destructuring and defaults**

   - Use destructuring for props
   - Provide default values where appropriate
   - Include prop spread (`...props`) to allow passing HTML attributes to the root element
   - **Do not use `Component.defaultProps`** — it is deprecated in React 19. Set defaults in the destructuring pattern instead:
     ```jsx
     export function MyComponent({ variant = 'primary', isLoading = false }) { ... }
     ```

3. **Prop naming conventions**

   Follow these conventions when naming props:

   | Category | Convention | Examples |
   |----------|-----------|---------|
   | General props | `camelCase` | `backgroundColor`, `imageSrc`, `imageAlt` |
   | Booleans | `is`/`has`/`show` prefix | `isLoading`, `hasError`, `showArrows` |
   | Event handlers | `on` prefix | `onClick`, `onPageChange`, `onClose` |
   | CSS class props | `className` | `className="custom-class"` |
   | Children content | `children` | Standard React children prop |

   Avoid generic names like `data` or `info` — use descriptive names that convey meaning (e.g., `chartData`, `userProfile`).

4. **Avoid module-level mutable state**

   Never use module-level variables (e.g., `Set`, `Map`, `Array`, counters) to track state across function calls. These persist between component instances and across renders, causing bugs that are difficult to diagnose. Always declare mutable tracking variables inside the function that uses them.

   ```jsx
   // Bad — accumulates across calls, leaks between instances
   const seenIds = new Set();
   export function processData(items) {
     items.forEach(item => seenIds.add(item.id));
   }

   // Good — fresh state per call
   export function processData(items) {
     const seenIds = new Set();
     items.forEach(item => seenIds.add(item.id));
   }
   ```

5. **Class naming conventions**

   - All component CSS classes should start with `mg-` prefix
   - Use kebab-case for class names (e.g., `mg-button-primary`)
   - Use BEM methodology for structuring CSS classes

6. **Semantic HTML**
   - Use the most semantically appropriate HTML elements
   - Choose elements based on their meaning, not their default styling
   - Follow these guidelines for common elements:
     - `<section>`: For thematic grouping of content with a heading
     - `<article>`: For self-contained, independently distributable content
     - `<nav>`: For navigation menus and links
     - `<aside>`: For content tangentially related to the main content
     - `<header>`, `<footer>`: For introductory/concluding content of sections
     - `<main>`: For the primary content of the document (only one per page)
     - `<figure>` and `<figcaption>`: For images, diagrams with captions
     - Use `<div>` only when no semantic element is appropriate
   - Always include appropriate ARIA attributes when needed
   - Use heading levels (`<h1>` through `<h6>`) in a logical, hierarchical order
   - Ensure interactive elements are keyboard accessible (use `<button>` for clickable elements, not styled `<div>`s)

### JSDoc documentation guidelines

All components must include comprehensive JSDoc comments to document their purpose, props, and usage. This improves code maintainability and enables better IDE autocompletion and type checking.

1. **Component Documentation**

   - Add a JSDoc comment block at the top of each component file
   - Include a description of the component's purpose and functionality
   - Document all props with proper types and descriptions
   - Example:

   ```jsx
   /**
    * Button component that follows Mangrove's component guidelines
    * @param {Object} props - Component props
    * @param {string} props.variant - Button variant ('primary', 'secondary', 'tertiary', 'danger')
    * @param {string} [props.size='md'] - Button size ('sm', 'md', 'lg')
    * @param {boolean} [props.isFullWidth=false] - Whether the button should take full width
    * @param {boolean} [props.isDisabled=false] - Whether the button is disabled
    * @param {Function} [props.onClick] - Click handler function
    * @param {React.ReactNode} props.children - Button content
    * @returns {JSX.Element} Rendered Button component
    */
   export function Button({
     variant,
     size = 'md',
     isFullWidth = false,
     isDisabled = false,
     onClick,
     children,
     ...props
   }) {
     // Implementation
   }
   ```

2. **Function Documentation**

   - Document helper functions and hooks with JSDoc comments
   - Specify parameter types, return types, and descriptions
   - Example:

   ```jsx
   /**
    * Formats a date according to the specified format
    * @param {Date} date - The date to format
    * @param {string} [format='MM/DD/YYYY'] - The format string
    * @returns {string} The formatted date string
    */
   function formatDate(date, format = 'MM/DD/YYYY') {
     // Implementation
   }
   ```

3. **Complex Props Documentation**
   - For components with complex nested props, document the structure clearly
   - Use array and object notation to indicate nested properties
   - Example:
   ```jsx
   /**
    * MegaMenu component that displays a navigation menu with sections and items
    * @param {Object[]} sections - Array of section objects containing menu structure
    * @param {string} sections[].bannerHeading - Heading text for the section banner
    * @param {string} sections[].bannerDescription - Description text for the section banner
    * @param {Object} sections[].bannerButton - Optional button object for the banner
    * @param {string} sections[].bannerButton.label - Label text for the banner button
    * @param {string} sections[].bannerButton.url - URL for the banner button
    * @param {Object[]} sections[].items - Array of menu items for the section
    * @param {string} sections[].items[].title - Title of the menu item
    * @param {string} sections[].items[].url - URL for the menu item
    * @param {Object[]} sections[].items[].items - Optional nested items
    * @param {number} [delay=300] - Delay in milliseconds before closing menu on mouse leave
    * @returns {JSX.Element} Rendered MegaMenu component
    */
   export function MegaMenu({ sections, delay = 300 }) {
     // Implementation
   }
   ```

### TypeScript components

Mangrove supports TypeScript for component development. Using TypeScript provides several benefits, including improved type safety, better IDE support, and enhanced documentation.

1. **TypeScript Component Structure**

   TypeScript components use the `.tsx` extension and follow the same general structure as JavaScript components:

   ```
   stories/Components/[ComponentCategory]/[ComponentName]/
   ├── ComponentName.tsx           # TypeScript component implementation
   ├── ComponentName.mdx           # Documentation
   ├── ComponentName.stories.tsx   # TypeScript Storybook stories
   └── component-name.scss         # Component-specific styles (optional)
   ```

2. **Declaring Props Types**

   Define component props using TypeScript interfaces or type aliases:

   ```tsx
   // Using type alias
   type ButtonProps = {
     label: string;
     type: string;
     width: number;
     onClick?: () => void; // Optional prop
   };

   // Component with typed props
   function TypedButton({ label, type, width, onClick }: ButtonProps) {
     return (
       <button
         className={`mg-button mg-button--${type}`}
         style={{ width: `${width}px` }}
         onClick={onClick}
       >
         {label}
       </button>
     );
   }
   ```

3. **TypeScript Stories**

   When writing stories for TypeScript components, use CSF3 object syntax with Storybook's `Meta` and `StoryObj` types:

   ```tsx
   import type { Meta, StoryObj } from '@storybook/react-webpack5';
   import TypedButton from './TypedButton';

   const meta: Meta<typeof TypedButton> = {
     title: 'Components/Buttons/TypedButton',
     component: TypedButton,
   };

   export default meta;
   type Story = StoryObj<typeof TypedButton>;

   // CSF3: use object syntax with args
   export const Primary: Story = {
     args: {
       label: 'Submit',
       type: 'primary',
       width: 120,
     },
   };
   ```

4. **Best Practices for TypeScript Components**

   - Use explicit return types for components (`React.FC` is discouraged)

   ```tsx
   // Preferred approach with explicit return type
   function Button(props: ButtonProps): JSX.Element {
     // Implementation
   }
   ```

   - Leverage TypeScript's union types for component variants

   ```tsx
   type ButtonVariant = 'primary' | 'secondary' | 'tertiary' | 'danger';

   type ButtonProps = {
     variant: ButtonVariant;
     // Other props
   };
   ```

   - Use type inference where appropriate but be explicit where it improves readability
   - Prefer interfaces for public APIs and type aliases for props and other internal types
   - Include proper JSDoc comments even with TypeScript types (they provide additional context)

5. **Example TypeScript component**

   The Mangrove library includes a TypeScript example component in `stories/Components/TypeScriptExampleComponent` that demonstrates these principles:

   ```tsx
   // TypeScriptExampleComponent.tsx
   import React from 'react';
   import { CtaButton } from '../Buttons/CtaButton/CtaButton';

   type ButtonProps = {
     /** Button label text */
     label: string;
     /** Visual style variant */
     type: 'Primary' | 'Secondary';
     /** Button width in pixels */
     width: number;
     /** Optional click handler */
     onClick?: () => void;
   };

   function TypeScriptExampleComponent({
     label,
     type,
     width,
     onClick,
   }: ButtonProps): JSX.Element {
     return (
       <CtaButton
         label={label}
         Type={type}
         State="Default"
         For_Primary="Text"
         variant="contained"
         style={{ width: `${width}px` }}
         onClick={onClick}
       >
         {label}
       </CtaButton>
     );
   }

   export default TypeScriptExampleComponent;
   ```

   And its CSF3 stories file:

   ```tsx
   // TypeScriptExampleComponent.stories.tsx
   import type { Meta, StoryObj } from '@storybook/react-webpack5';
   import TypeScriptExampleComponent from './TypeScriptExampleComponent';

   const meta: Meta<typeof TypeScriptExampleComponent> = {
     title: 'Example/TypeScript component',
     component: TypeScriptExampleComponent,
   };

   export default meta;
   type Story = StoryObj<typeof TypeScriptExampleComponent>;

   export const Submit: Story = {
     args: {
       label: 'Submit',
       type: 'Secondary',
       width: 400,
     },
   };
   ```

### Styling guidelines

1. **Use SCSS for component styles**

   - Each component may have its own SCSS file
   - Import shared variables and mixins from the global styles

2. **Follow the BEM (Block, Element, Modifier) methodology**

   - **Block**: The main component (e.g., `.mg-card`)
   - **Element**: A part of the block (e.g., `.mg-card__title`, `.mg-card__content`)
   - **Modifier**: A variation of a block or element (e.g., `.mg-card--featured`, `.mg-card__title--large`)
   - Use double underscores (`__`) to separate block from elements
   - Use double hyphens (`--`) to separate block/element from modifiers
   - Example structure:

     ```scss
     .mg-card {
       // Block styles

       &__title {
         // Element styles

         &--large {
           // Modifier styles
         }
       }

       &--featured {
         // Block modifier styles
       }
     }
     ```

   - Avoid deep nesting of elements (max 2-3 levels)
   - Keep selectors as flat as possible for better performance
   - Don't use element tags in selectors (e.g., use `.mg-card__title` not `h2.mg-card__title`)

3. **Follow the existing design tokens**

   - Use variables for colors, spacing, typography, etc.
   - Retrieve design tokens from `stories/assets/scss/variables`
   - Never use hardcoded values for colors, fonts, or spacing
   - Example:
     ```scss
     .mg-button {
       background-color: $mg-color-interactive; // Good
       padding: $mg-spacing-md $mg-spacing-lg; // Good
       // NOT background-color: #0066cc; // Bad - hardcoded value
     }
     ```

4. **Responsive design**

   - All components must be responsive
   - Use the breakpoint mixins provided in the project
   - Design mobile-first, then add styles for larger screens
   - Example:

     ```scss
     .mg-component {
       // Mobile styles (default)

       @include devicebreak(medium) {
         // Tablet styles (≥ 48em)
       }

       @include devicebreak(large) {
         // Desktop styles (≥ 64em)
       }
     }
     ```

     Available breakpoints: `small` (≤ 47.9375em), `medium` (≥ 48em), `large` (≥ 64em), `xlarge` (≥ 90em), `mediumonlytab` (≤ 89.9375em). See `stories/assets/scss/_breakpoints.scss` for definitions.

5. **Accessibility**
   - Ensure sufficient color contrast (WCAG AA minimum)
   - Support keyboard navigation and screen readers
   - Test with accessibility tools
   - Use appropriate focus states for interactive elements

## RTL support

UNDRR websites serve content in Arabic and other right-to-left (RTL) languages. Every component must work correctly in both LTR and RTL layouts.

### How RTL is triggered

The Storybook locale toolbar (globe icon) controls text direction. Selecting **Arabic** sets `dir="rtl"` and `lang="ar"` on the document and on the `#storybook-root` element. The runtime globals `window.UNDRR.langCode` and `window.UNDRR.dir` are also updated. Selecting any other locale resets to `dir="ltr"`.

> **Note:** `window.UNDRR.langCode` stores the English-language name of the locale (e.g., `'arabic'`, `'english'`, `'burmese'`, `'japanese'`), not an ISO 639-1 code. To get the ISO code (e.g., `'ar'`, `'en'`), use the `langArr` mapping in `.storybook/preview.js` or the `lang` attribute on the `<html>` element.

This logic lives in `.storybook/preview.js` in the `getLangCode` decorator. There is also a legacy manual RTL toggle (`setDirection` decorator) that can override the locale setting.

### SCSS patterns for RTL

There are two approaches for handling directional layout in SCSS. Prefer logical properties for new code; fall back to `[dir="rtl"]` overrides when logical properties are not sufficient.

#### 1. CSS logical properties (preferred)

Use logical properties instead of physical ones. These automatically flip in RTL contexts with no extra CSS:

| Physical property | Logical replacement |
|---|---|
| `margin-left` / `margin-right` | `margin-inline-start` / `margin-inline-end` |
| `padding-left` / `padding-right` | `padding-inline-start` / `padding-inline-end` |
| `left` / `right` | `inset-inline-start` / `inset-inline-end` |
| `text-align: left` | `text-align: start` |
| `float: left` | `float: inline-start` |

Shorthand forms like `margin-inline`, `padding-inline`, and `inset-inline` set both sides at once. See `megamenu.scss` for extensive use of this approach.

#### 2. `[dir="rtl"]` selector overrides

When logical properties cannot express the override (e.g., flipping a `transform`, changing a `background-position`, or mirroring an icon), use the `[dir="rtl"]` attribute selector:

```scss
.mg-component__arrow {
  transform: rotate(0deg);
}

[dir="rtl"] .mg-component__arrow {
  transform: rotate(180deg);
}
```

Or nest with `&` when the override is scoped to the block:

```scss
.mg-component__separator {
  border-left: 2px solid $mg-color-neutral-400;

  [dir="rtl"] & {
    border-left: none;
    border-right: 2px solid $mg-color-neutral-400;
  }
}
```

The `_mixins.scss` file includes RTL-aware placeholders such as `%select_rtl` (padding/text-align flip for select inputs), `%select_rtl_after` (repositions a pseudo-element for RTL), and `%left-to-right` (entrance animation that reverses direction under `[dir="rtl"]`).

#### RTL patterns in global styles

- `_foundational.scss` sets `direction: rtl` on `ul` and `ol` elements inside `[dir="rtl"]`
- `_utility.scss` reverses viewport entrance animations under `[dir="rtl"]`

### Components with RTL stories

These components have dedicated RTL story exports you can reference as examples:

- **MegaMenu** — `RTL` story: full menu layout with flipped chevrons, borders, and sidebar
- **Pager** — `RTL` story: pagination controls in Arabic
- **QuoteHighlight** — `RTLLeftAlignedQuote`, `RTLRightAlignedQuote`: directional float behavior
- **HighlightBox** — `RTLFloatStart`, `RTLFloatEnd`: logical float behavior in RTL
- **IndexChart** — `CustomRtlIndexChart`: chart axis and label direction
- **ScatterPlotChart** — `CustomRtlConnectedScatterplot`: RTL chart rendering
- **StatsCard** — locale-aware story that switches direction based on toolbar selection

Many other components (Breadcrumbs, Buttons, Cards, Forms, Tab, Table, Hero) include Arabic translations that respond to the locale toolbar without needing a separate RTL story export.

### Checklist for RTL-compatible components

When building or reviewing a component, verify each of these:

- [ ] Use `margin-inline-*`, `padding-inline-*`, `inset-inline-*` instead of physical `left`/`right` equivalents
- [ ] Use `text-align: start` / `text-align: end` instead of `left` / `right`
- [ ] Flip arrows, chevrons, and carets under `[dir="rtl"]` (typically by reversing a `transform`)
- [ ] Swap `border-left` / `border-right` under `[dir="rtl"]`, or use `border-inline-start` / `border-inline-end`
- [ ] Verify `flex-direction: row` reverses correctly in RTL; avoid forcing `row-reverse`
- [ ] Replace positional `left` / `right` with `inset-inline-start` / `inset-inline-end`
- [ ] Reverse `translateX` and `rotate` values under `[dir="rtl"]` where direction matters
- [ ] Avoid inline styles with `left`, `right`, `marginLeft`, `marginRight` (these do not respond to `dir`)

### Testing with the locale toolbar

1. Start Storybook with `yarn dev`
2. Open your component's story
3. Click the globe icon in the Storybook toolbar
4. Select **Arabic** to switch to RTL mode
5. Verify the layout mirrors correctly: text flows right-to-left, icons/chevrons point the opposite direction, spacing and borders appear on the correct side
6. Switch back to **English** to confirm no layout artifacts remain

If your component has complex RTL behavior, add a dedicated RTL story export (see the MegaMenu or QuoteHighlight stories for the pattern).

## Documentation requirements

Each component must have comprehensive documentation in an MDX file that includes:

1. **Overview** - Brief description of what the component is
2. **When to use** - Guidance on when this component should be used
3. **Formatting** - Default and variant appearances
4. **Content** - Guidelines for content within the component
5. **Behaviors** - How the component responds to interaction
6. **CSS and JS References** - Required files to include
7. **Interactions** - Details on hover, focus, and other states
8. **Changelog** - Version history of the component

### Changelog format

Every component MDX file must end with a `## Changelog` section (H2 level) that tracks the component's version history. Use this exact format:

```markdown
## Changelog

- **2.1** — 2026-02-20:
  - ([web-backlog#1992](https://gitlab.com/undrr/web-backlog/-/issues/1992)) Taxonomy term results
  - ([web-backlog#2636](https://gitlab.com/undrr/web-backlog/-/issues/2636)) Card display modes
- **2.0** — 2026-01-15 ([web-backlog#1234](https://gitlab.com/undrr/web-backlog/-/issues/1234)): **Breaking:** Description of breaking change
- **1.0** — 2025-11-01 ([web-backlog#999](https://gitlab.com/undrr/web-backlog/-/issues/999)): Released component
```

**Rules:**

- **Heading level**: Always `## Changelog` (H2)
- **Single-issue entry**: `- **version** — YYYY-MM-DD ([web-backlog#NNN](url)): description`
- **Multi-issue entry**: When multiple issues ship in the same version, use sub-bullets — one per issue, each citing its own tracker issue
- **Tracker issue**: Always cite the GitLab tracker issue in parentheses using the short format `web-backlog#NNN` linked to the full URL
- **Version**: Semver without `v` prefix (e.g., `1.0`, `2.1.3`, not `v1.0` or `Version 1.0`)
- **Separator**: Em dash `—` between version and date, colon `:` after date
- **Date**: ISO 8601 format (`YYYY-MM-DD`) for traceability
- **Order**: Newest entries first
- **Breaking changes**: Prefix description with `**Breaking:**`
- **Structure**: Bullet list with bold version number (compact, scannable)
- **Description**: Summarize the user-facing change, not implementation details. Use sentence case

**Do not** use subheadings within changelog entries, date-only entries without version numbers, bare version numbers without descriptions, or mixed separator styles.

### Title capitalization rules

All titles and headings in the Mangrove documentation follow **sentence case capitalization**:

- **Capitalize only the first word** of titles and headings
- **Capitalize proper nouns** (names, places, technologies, brands)
- **Use lowercase for all other words**

This rule applies to:

- Main page headlines (H1)
- Section headings (H2, H3, H4, etc.)
- Meta title attributes in MDX files
- Navigation titles in Storybook

**Examples:**

✅ **Correct:**

- `# Getting started with mangrove`
- `## Component structure and standards`
- `### CSS and JS references`
- `title="Getting started/React integration"`
- `# TypeScript demo component`

❌ **Incorrect:**

- `# Getting Started With Mangrove`
- `## Component Structure And Standards`
- `### CSS And JS References`
- `title="Getting Started/React Integration"`
- `# TypeScript Demo Component`

**Exceptions for proper nouns:**

✅ **Keep these capitalized:**

- Technology names: React, TypeScript, JavaScript, HTML, CSS, Sass, SCSS
- Framework/library names: Storybook, Jest, UNDRR, Mangrove
- Acronyms: API, URL, RTL, LTR, WCAG, MDX
- Brand names: GitHub, GitLab, Google Analytics

**Example with proper nouns:**

- `# React integration guide` ✅
- `## Working with non-TypeScript code` ✅
- `### CSS and JS references` ✅
- `# UNDRR logos` ✅

This approach ensures consistency across all documentation and provides a clean, readable hierarchy that matches modern documentation standards.

Example structure:

```mdx
import { Meta, Story, Canvas } from '@storybook/addon-docs/blocks';
```

## Creating Storybook stories

Stories demonstrate the component's usage and variants:

1. **Basic structure (CSF3 format)**

   All stories must use **CSF3 object syntax**. Do not use the legacy `Template.bind({})` pattern.

   The `component` property is **required** in the default export — it enables autodocs prop tables and type inference.

   ```jsx
   import { MyComponent } from './MyComponent';

   export default {
     title: 'Components/Category/ComponentName',
     component: MyComponent, // Required for autodocs
     argTypes: {
       variant: {
         options: ['primary', 'secondary'],
         control: { type: 'radio' },
       },
     },
   };

   // CSF3: use object syntax with args
   export const Default = {
     args: {
       variant: 'primary',
     },
   };

   // Use render only when you need custom JSX wrapping
   export const WithWrapper = {
     args: { variant: 'secondary' },
     render: args => (
       <div style={{ padding: '1rem' }}>
         <MyComponent {...args} />
       </div>
     ),
   };
   ```

2. **Include multiple variants**

   - Create separate story exports for each significant variant
   - Show edge cases and different states

3. **Internationalization**
   - Support both LTR and RTL text directions
   - Include examples with different languages when relevant

## Testing requirements

Components should be thoroughly tested to ensure stability and reliability.

### Unit tests

1. **Create test files**

   - Tests should be placed in a `__tests__` folder within your component directory or at the appropriate level in the project structure
   - Follow the naming pattern: `ComponentName.test.jsx`

2. **Required tests**
   - Rendering test (component renders without errors)
   - Props test (component correctly responds to different props)
   - Event tests (click handlers and other events work correctly)
   - Accessibility tests (component meets accessibility requirements)

Example test pattern:

```jsx
import React from 'react';
import { render, screen, fireEvent } from '@testing-library/react';
import { axe } from 'jest-axe';
import { MyComponent } from '../MyComponent';

describe('MyComponent', () => {
  const defaultProps = {
    variant: 'primary',
    label: 'Click me',
  };

  it('renders without crashing', () => {
    render(<MyComponent {...defaultProps} />);
    expect(screen.getByText('Click me')).toBeInTheDocument();
  });

  it('responds to props correctly', () => {
    render(<MyComponent {...defaultProps} variant="secondary" />);
    // Assert component renders with secondary variant
  });

  it('handles events correctly', () => {
    const handleClick = jest.fn();
    render(<MyComponent {...defaultProps} onClick={handleClick} />);
    fireEvent.click(screen.getByRole('button'));
    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  it('has no a11y violations', async () => {
    const { container } = render(<MyComponent {...defaultProps} />);
    expect(await axe(container)).toHaveNoViolations();
  });
});
```

### Running tests

Use the following commands to run tests:

```bash
# Run all tests
yarn test

# Run tests with coverage report
yarn test:coverage

# Run tests in watch mode during development
yarn test:watch
```

## Deprecating components

When a component is no longer needed, has been replaced, or has been merged into another component, follow this process to deprecate and eventually remove it.

### When to deprecate

- The component has been **replaced** by a newer component with better functionality
- The component is **no longer needed** due to changes in design or requirements
- The component has been **merged** into another component

### How to mark a component as deprecated

Apply all three of the following:

1. **Console warning** — Add a runtime warning so consumers are notified during development:

   ```jsx
   export function OldComponent(props) {
     React.useEffect(() => {
       console.warn(
         'OldComponent is deprecated and will be removed in the next major version. Use NewComponent instead.'
       );
     }, []);
     // ...
   }
   ```

2. **JSDoc `@deprecated` tag** — Add to the component's JSDoc block so IDEs surface the deprecation:

   ```jsx
   /**
    * @deprecated Use NewComponent instead. Will be removed in the next major version.
    */
   export function OldComponent(props) { ... }
   ```

3. **Storybook badge** — Add a `deprecated` tag and a banner in the component's MDX documentation:

   ```jsx
   // ComponentName.stories.jsx
   export default {
     title: 'Components/Category/ComponentName',
     tags: ['deprecated'],
   };
   ```

   ```mdx
   {/* ComponentName.mdx */}
   > **Deprecated:** This component will be removed in the next major version.
   > Use [NewComponent](?path=/docs/components-category-newcomponent--docs) instead.
   ```

### Migration path

Every deprecation must include a clear migration path:

- Document the replacement component or alternative approach in the MDX file
- Link directly to the replacement component's Storybook page
- Note any prop name changes or behavioral differences between old and new

### Timeline

- Deprecated components must remain available for **at least one minor release** before removal
- Removal is a **breaking change** and requires a major version bump

### Changelog entry

Use the standard changelog format with a `**Deprecated:**` prefix:

```markdown
- **2.1** — 2026-03-01: **Deprecated:** Use NewComponent instead
```

### Removal

When removing a deprecated component:

- Use `BREAKING CHANGE:` in the commit body to trigger a major version bump
- Remove all related files (component, stories, styles, tests, MDX)
- Update any cross-references in other component documentation

## Troubleshooting common issues

### Component styling issues

- Check that your SCSS files are properly imported
- Verify CSS class names follow the naming conventions
- Inspect elements using browser developer tools to identify CSS conflicts

### Storybook display problems

- Ensure your component export is correctly set up
- Check that your stories file properly imports the component
- Verify the Meta and Canvas tags in your MDX file are correctly formatted

### Testing failures

- Check that all dependencies are correctly installed
- Ensure test files are in the correct location
- Verify that components are correctly imported in test files

### Build errors

- Make sure all required dependencies are installed
- Check for syntax errors in your JSX/JavaScript
- Verify that your SCSS is valid and properly formatted
